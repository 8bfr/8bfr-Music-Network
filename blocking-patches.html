Blocking Filter — Patch Snippets

Deploy order: (1) Run user-blocks.sql in Supabase first, (2) deploy new profile.html, (3) deploy new feed.html (already patched), (4) apply snippets below to dm.html and notifications.html.

OWNER_ID constant used everywhere: cb556180-f032-4b21-9470-1d786f2664ab

1. dm.html — block filter for message list + inbox

Step 1: Add blockedIds state at the top of your script

var OWNER_ID='cb556180-f032-4b21-9470-1d786f2664ab';
var blockedIds=[]; // users to hide from DM inbox and search

Step 2: Load blocks right after you load the current user

// After you have currentUser (cu / currentUser / user), add:
if(currentUser){
  try{
    var mb=await sb.from('user_blocks').select('blocked_id').eq('blocker_id',currentUser.id);
    (mb.data||[]).forEach(function(b){ if(blockedIds.indexOf(b.blocked_id)===-1) blockedIds.push(b.blocked_id); });
    if(currentUser.id!==OWNER_ID){
      var bb=await sb.from('user_blocks').select('blocker_id').eq('blocked_id',currentUser.id);
      (bb.data||[]).forEach(function(b){ if(blockedIds.indexOf(b.blocker_id)===-1) blockedIds.push(b.blocker_id); });
    }
  }catch(e){}
}

Step 3: Filter the DM inbox query

Wherever you query direct_messages to build the inbox list, add the block filter:

// BEFORE:
var r = await sb.from('direct_messages')
  .select('*')
  .or('sender_id.eq.'+cu.id+',receiver_id.eq.'+cu.id)
  .order('created_at',{ascending:false});

// AFTER (add the .not filter for each side):
var q = sb.from('direct_messages')
  .select('*')
  .or('sender_id.eq.'+cu.id+',receiver_id.eq.'+cu.id)
  .order('created_at',{ascending:false});
if(blockedIds.length){
  q = q.not('sender_id','in','('+blockedIds.join(',')+')')
       .not('receiver_id','in','('+blockedIds.join(',')+')');
}
var r = await q;

Step 4: Block sending to blocked users (defensive)

Before inserting a new DM, check:

// Right before sb.from('direct_messages').insert(...)
if(blockedIds.indexOf(receiverId)!==-1){
  alert('You cannot message this user.');
  return;
}

Step 5: Filter user picker / search

If you have a "start new message" user search, filter results client-side:

// After you get search results
results = (results||[]).filter(function(u){
  return blockedIds.indexOf(u.user_id)===-1;
});

2. notifications.html — hide notifications from blocked users

Step 1: Same blockedIds loader as above

Add the same OWNER_ID, blockedIds variable and the loader block right after you get the current user.

Step 2: Filter the notifications query

// BEFORE:
var r = await sb.from('notifications')
  .select('*')
  .eq('user_id', cu.id)
  .order('created_at',{ascending:false});

// AFTER:
var q = sb.from('notifications')
  .select('*')
  .eq('user_id', cu.id)
  .order('created_at',{ascending:false});
if(blockedIds.length){
  // Hide notifications triggered by blocked users
  q = q.not('from_user_id','in','('+blockedIds.join(',')+')');
}
var r = await q;

Note: If your notifications table uses a different column name (actor_id, sender_id, source_user_id), replace from_user_id with the correct column name. Check with:
SELECT column_name FROM information_schema.columns WHERE table_name='notifications';

3. Comments on feed posts — hide comments from blocked users

In feed.html (and any other page that loads comments), find your comments query and add the filter. The pattern is the same:

// Anywhere you load comments:
var q = sb.from('comments').select('*').eq('post_id', postId).order('created_at');
if(blockedIds.length){
  q = q.not('user_id','in','('+blockedIds.join(',')+')');
}
var r = await q;

4. Optional: use the SQL helper function instead

If you prefer server-side filtering, you can call get_blocked_user_ids():

var bl = await sb.rpc('get_blocked_user_ids', { uid: cu.id });
var blockedIds = (bl.data||[]).map(function(r){ return r.user_id; });

This handles the owner-exception automatically. Use either this approach or the two-query client-side approach — not both.
